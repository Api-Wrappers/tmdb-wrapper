TMDB Wrapper Improvement Audit

Created: 2026-06-25 · Status: ready · Target: @api-wrappers/tmdb-wrapper

TL;DR

The wrapper is already in a good state: strict TypeScript, clear endpoint groups, useful docs, package build tests, and path-mapping tests. The best next improvement is not a rewrite; it is a focused internal cleanup that makes all endpoint groups share one transport client and standardizes per-request config support across endpoint methods. After that, tighten type exports and docs/tests so the public package is easier to trust and maintain.

1. Goal

Improve the API wrapper in ways that make it safer, easier to maintain, and more reliable for package consumers without changing the current public new TMDB(...) facade or endpoint-group shape.

2. Codebase Understanding

  • Stack: TypeScript ESM package, Bun for local scripts/tests, tsdown for ESM/CJS/declarations, Biome for checks, @api-wrappers/api-core for transport.
  • Package surface: package.json:5-15 exports only the root entrypoint with ESM, CJS, and declaration files.
  • Facade: src/index.ts:30-84 exposes a TMDB class with 25 endpoint-group properties.
  • Transport: src/client.ts:23-89 wraps BaseHttpClient, applies TMDB auth, defaults retries, and checks auth before fetching.
  • Endpoint base: src/@types/models/baseEndpoint.ts:9-24 constructs a new TMDBApiClient for each endpoint instance.
  • Request config pattern: newer endpoints use RequestConfig and withQuery from src/utils/query.ts:25-31, but older endpoints still pass raw { query: options } and do not expose request config consistently.
  • Tests: tests/client.test.ts covers auth, retry, errors, and query serialization. tests/endpoint-coverage.test.ts verifies many endpoint paths, query parameters, methods, and bodies. tests/package-build.test.ts verifies built entrypoints.
  • Validation observed: bun run typecheck passed. bun run check passed with 64 files checked.

3. Recommendation

Do this now as one focused execution task. The codebase is healthy enough that a broad rewrite would waste effort. The first high-value task should be transport sharing plus request-config parity because it improves runtime behavior, removes duplication, and brings the code in line with the docs claim that most endpoint methods accept request overrides.

Do not generate a large TMDB client or redesign the public endpoint API yet. Preserve the current manual endpoint style and add coverage incrementally.

Priority Findings

High: Each endpoint group creates its own HTTP client

Evidence: TMDB constructs every endpoint independently in src/index.ts:57-83. BaseEndpoint then creates a fresh TMDBApiClient in src/@types/models/baseEndpoint.ts:22-24.

Why it matters: One new TMDB(config) currently produces many transport instances. That duplicates normalized auth, default headers, retry config, and injected fetch handling. If api-core caching, plugins, telemetry, or per-client state are used, those behaviors are fragmented by endpoint group.

Recommended fix: Create one TMDBApiClient in TMDB and inject it into all endpoint groups through BaseEndpoint. Keep endpoint constructors source-compatible where practical.

High: Request-config support is inconsistent across endpoint methods

Evidence: docs/request-config.md:1-4 says most endpoint methods accept RequestConfig. MoviesEndpoint.details accepts request overrides in src/endpoints/movies.ts:69-80, while TvShowsEndpoint.details, alternativeTitles, changes, contentRatings, and aggregateCredits do not in src/endpoints/tvShows.ts:69-142. GenreEndpoint and ChangeEndpoint also pass raw query objects without request config in src/endpoints/genre.ts:30-43 and src/endpoints/changes.ts:34-59.

Why it matters: Consumers cannot reliably use timeout, abort signal, custom headers, cache key, or tags across endpoint groups. The wrapper’s own docs create an expectation the API does not uniformly satisfy.

Recommended fix: Standardize endpoint signatures so every network method accepts optional RequestConfig as its final parameter, then route all query-bearing calls through withQuery.

Medium: Query helper uses casts that can hide shape drift

Evidence: src/utils/query.ts:25-31 casts arbitrary objects to QueryParams. src/client.ts:68-85 casts request options and query values again.

Why it matters: The project is strict and avoids any, but repeated casts make it easier for endpoint option types to diverge from what api-core can serialize.

Recommended fix: Introduce a narrow local endpoint-query type and helper that accepts only serializable primitives and arrays used by api-core. Keep the implementation tiny and do not add runtime validation unless tests expose a real need.

Medium: Published package has only a single root export

Evidence: package.json:9-15 exposes only ".". The README imports both TMDB and helpers from the root, but consumers cannot import subpaths such as image utilities or types directly.

Why it matters: A root-only export is simple, but it makes tree-friendly imports and docs navigation less ergonomic as the wrapper grows.

Recommended fix: Keep the root export, then consider adding stable subpath exports for ./utils and ./types only after the first transport/request-config cleanup is complete. This may require tsdown entry changes and package-build tests.

Low: Roadmap and docs already identify useful next examples, but should follow source parity work

Evidence: ROADMAP.md already calls out endpoint docs alignment, request-config tests, auth docs, and examples. docs/contributing-ideas.md:5-19 lists approachable docs/test improvements.

Why it matters: Docs are good enough to guide users, but docs improvements should not get ahead of inconsistent API signatures.

Recommended fix: After source parity lands, update docs examples for timeout/abort, API key versus token auth, TV season/episode usage, and account watchlists.

4. Scope

In Scope

  • Refactor endpoint construction so one TMDBApiClient is shared across all endpoint groups.
  • Standardize RequestConfig as the last argument on endpoint methods that perform network calls.
  • Replace raw { query: options } call sites with withQuery(options, request) where request config is added.
  • Add or extend tests that prove shared fetch/client behavior and request-config parity.
  • Update docs only where they describe request config, auth behavior, or changed examples.

Out of Scope

  • Do not rewrite the endpoint model around generated code.
  • Do not remove the root TMDB class or rename existing endpoint groups.
  • Do not add new TMDB endpoint coverage in the same task unless needed for tests.
  • Do not change auth semantics beyond preserving current API key and bearer-token behavior.
  • Do not add dependencies.

6. Implementation Plan

  1. Characterize current behavior. Run the existing targeted tests before editing: bun run typecheck, bun run check, bun test tests/client.test.ts tests/tmdb-client.test.ts tests/endpoint-coverage.test.ts. Do not run format write commands before making source changes.
  2. Share the transport client. In src/index.ts, create one TMDBApiClient in the TMDB constructor. Update BaseEndpoint so endpoint classes receive that client instead of constructing their own. Preserve endpoint class property names and public new TMDB(auth) usage.
  3. Keep endpoint constructors boring and compatible. Update endpoint constructors in src/endpoints/*.ts to call super(api) or use a small constructor helper pattern. Avoid per-endpoint custom transport logic.
  4. Add a shared-client regression test. In tests/tmdb-client.test.ts or tests/client.test.ts, inject a custom fetch through client.fetch, call methods on multiple endpoint groups, and assert the same fetch configuration is honored across those groups. If direct identity checks require exposing internals, avoid exposing internals and verify behavior through calls.
  5. Standardize request-config signatures incrementally. Sweep src/endpoints/*.ts for methods that call this.api.get, post, delete, or similar without a final request?: RequestConfig. Add request?: RequestConfig as the last parameter and pass it through with withQuery or directly when no query exists.
  6. Use withQuery consistently. Replace raw { query: options } with withQuery(options, request) once the method accepts request config. For computed query objects such as append-to-response or image-language arrays, keep the existing csv behavior.
  7. Protect public compatibility. For methods where adding a final optional argument could conflict with an existing positional API, preserve current parameter order and only append the request argument. Do not reorder existing options, IDs, session IDs, or bodies.
  8. Tighten query typing only if it stays small. If the casts in src/utils/query.ts can be reduced with a local EndpointQuery type compatible with api-core, do that in the same pass. If it becomes invasive, leave it for a second task and document why.
  9. Expand endpoint coverage tests for request config. Add focused tests that call representative methods across older and newer endpoint groups with { headers, timeoutMs, signal, cacheKey, tags } where observable. At minimum verify custom headers or abort signal pass to fetch for movies, TV shows, genre, changes, and account/search-style query endpoints.
  10. Update docs after tests pass. Update docs/request-config.md and README snippets only if examples need changed argument placement or if request-config support becomes materially broader. Keep docs source-verified.
  11. Run final validation. Run bun run typecheck, bun run check, bun test, and bun run build. If preparing a release, run bun pm pack --dry-run or bun run verify.

7. Code Quality Rules

  • Follow the existing manual endpoint-class style.
  • Keep the public TMDB facade stable.
  • Avoid duplicate transport setup.
  • Do not weaken TypeScript strictness.
  • Do not use any.
  • Prefer existing RequestConfig, csv, and withQuery helpers before adding new helpers.
  • Avoid adding dependencies.
  • Keep docs claims tied to implemented and tested behavior.

8. Testing and Validation

  • bun run typecheck before and after changes.
  • bun run check before and after changes.
  • bun test tests/client.test.ts tests/tmdb-client.test.ts tests/endpoint-coverage.test.ts while iterating.
  • bun test before finishing.
  • bun run build before finishing because package declarations and CJS/ESM outputs are part of the public contract.
  • bun pm pack --dry-run if the task includes package export changes.

9. Acceptance Criteria

  • A TMDB instance uses one shared configured TMDBApiClient across all endpoint groups.
  • Existing examples such as new TMDB(token), tmdb.movies.details(...), and tmdb.search.movies(...) continue to compile.
  • Endpoint methods consistently support final-argument RequestConfig unless there is a documented exception.
  • Raw query-object call sites are replaced with the shared query helper where request config is involved.
  • Tests cover shared transport behavior and representative request-config pass-through across multiple endpoint groups.
  • bun run typecheck, bun run check, bun test, and bun run build pass.
  • Docs remain accurate and do not claim unsupported endpoint coverage.

10. Risks and Notes

  • Compatibility risk: Endpoint constructor signatures are internal today, but classes are exported. If consumers instantiate endpoint classes directly, changing constructors may be a breaking change. Consider preserving overloads or documenting that endpoint classes are intended for the TMDB facade.
  • Behavior risk: Sharing one transport client may expose stateful api-core behavior across endpoint groups. This is likely desirable, but tests should verify auth, default headers, retry, and custom fetch still behave as expected.
  • Scope risk: Sweeping all endpoints can become large. If the diff grows too much, split into two PRs: shared client first, request-config parity second.
  • Docs risk: Do not update docs before source behavior is confirmed by tests.
  • Package risk: Subpath exports are useful but should be a later task unless the executor is explicitly asked to handle package exports.
